# Segfault Shrine User Flow

## 1. 核心用户路径

Segfault Shrine 是一个单页 Web 应用，用户应在一个连续页面中完成从输入报错到获得占卜结果的完整流程。

核心路径：

1. 用户打开页面，看到项目标题、标语、报错输入框和抽牌按钮。
2. 用户输入报错信息、日志片段或问题描述。
3. 用户点击“抽牌”。
4. 系统校验输入内容。
5. 如果输入为空，页面显示空输入提示，用户留在输入状态。
6. 如果输入有效，页面进入抽牌中状态。
7. 系统根据关键词和随机规则抽出三张不同的 Debug 塔罗牌。
8. 三张牌以 3D 翻牌效果揭晓。
9. 页面展示三张牌和本次 Debug 占卜解读。
10. 用户可以复制结果、生成截图或重新抽牌。

## 2. 页面列表

MVP 只有一个页面：

- 首页 / 占卜页

该页面包含多个状态：

- 初始输入状态。
- 空输入提示状态。
- 抽牌中状态。
- 抽牌结果状态。
- 截图生成失败状态。

不新增独立详情页、图鉴页、历史页、登录页或设置页。

## 3. 页面目标

首页 / 占卜页的目标：

- 快速建立“代码错误祭坛”的仪式感。
- 让用户明确知道可以输入报错并抽牌。
- 让抽牌结果具有足够强的视觉记忆点。
- 让用户能轻松复制或生成截图分享结果。
- 在桌面端和移动端都能完成核心流程。

## 4. 页面主要模块

### 4.1 顶部标题区

目标：

- 让用户第一眼理解项目气质和主题。

内容：

- 项目名：`Segfault Shrine`。
- 标语：一句短句，表达“用玄学解决玄学 Bug”的语气。
- 可选小标签：例如 `Debug Tarot`、`Cyber Occult`、`No API, only fate`。

规则：

- 项目名必须是第一屏的强视觉信号。
- 不做普通 SaaS landing page 的长篇介绍。
- 不放复杂导航栏，MVP 没有多页面导航。

### 4.2 报错输入区

目标：

- 收集用户的报错信息，为抽牌和解读提供上下文。

内容：

- 多行文本输入框。
- 占位提示，例如：“在此献祭你的报错日志...”
- 可选辅助文字，说明可以输入报错、堆栈、日志或一句问题描述。

交互：

- 用户可以粘贴多行日志。
- 输入为空时点击抽牌，输入框进入错误提示状态。
- 抽牌中输入框不可编辑或视觉上弱化。
- 抽牌结果出现后，默认保留用户输入，方便重新抽牌或修改后再抽。

### 4.3 抽牌操作区

目标：

- 提供核心仪式触发点。

内容：

- 主按钮：`抽牌` / `Draw Cards`。

交互：

- 输入有效时点击按钮，进入抽牌中状态。
- 输入为空时点击按钮，显示空输入提示。
- 抽牌中按钮不可重复点击。
- 结果出现后，主操作变为 `重新抽牌` 或在结果操作区展示 `重新抽牌`。

### 4.4 卡牌展示区

目标：

- 展示三张 Debug 塔罗牌，是页面最重要的视觉中心。

内容：

- 三张不同的牌。
- 每张牌展示牌名、关键词、图案区域和简短说明。
- 三张牌分别代表三个不同 Bug 线索。

交互：

- 抽牌中先显示卡背或占位牌。
- 结果出现时三张牌执行 3D 翻牌效果。
- 桌面端可支持 hover 轻微倾斜或高光，但不作为 MVP 必须能力。
- 移动端不依赖 hover，必要交互使用 tap。

规则：

- 三张牌不能重复。
- 卡牌必须足够清晰，不能被特效遮挡。
- 卡牌区域不承担图鉴或详情页功能。

### 4.5 解读结果区

目标：

- 将输入内容和三张牌串联成一段有技术梗的神秘学 Debug 解读。

内容：

- 本次占卜标题。
- 输入报错的短摘录。
- 三张牌的组合解读。
- 一段荒诞但像那么回事的 Debug 建议。

规则：

- 解读由本地模板生成，不调用 AI API。
- 文案风格可以夸张，但需要围绕 Debug、依赖、缓存、类型、环境、祖传代码等程序员语境。
- 解读文本应适合复制和截图分享。

### 4.6 结果操作区

目标：

- 让用户完成分享或再次游玩。

按钮：

- `重新抽牌`
- `复制结果`
- `生成截图`

交互：

- `重新抽牌`：保留当前输入内容，重新抽三张牌并生成新解读。
- `复制结果`：复制包含项目名、输入摘录、三张牌名和解读文案的纯文本。
- `生成截图`：将指定结果区域渲染为图片，供用户保存或分享。

规则：

- 操作按钮不应抢占卡牌和解读的视觉重点。
- 移动端按钮需要易点击，避免太小或太靠近屏幕边缘。

## 5. 主要状态

### 5.1 初始输入状态

触发条件：

- 用户首次打开页面。
- 用户尚未完成抽牌。

页面表现：

- 标题区可见。
- 输入框为空或显示占位提示。
- 抽牌按钮可见。
- 卡牌区可以显示三张卡背、符号占位或低透明度轮廓。
- 解读区可以隐藏，或显示等待占卜的短提示。

用户可做：

- 输入报错。
- 点击抽牌。

### 5.2 空输入提示状态

触发条件：

- 用户未输入内容时点击抽牌。

页面表现：

- 输入框边框或发光效果变为错误状态。
- 显示短提示，例如：“未发现可献祭的代码。”
- 页面不进入抽牌中状态。

用户可做：

- 输入内容后再次点击抽牌。

### 5.3 抽牌中状态

触发条件：

- 用户输入有效内容并点击抽牌。

页面表现：

- 抽牌按钮进入 loading 或 disabled 状态。
- 输入区视觉弱化，避免用户误以为可以继续编辑。
- 卡牌区显示卡背、洗牌、代码扫描或法阵仪式动效。
- 可显示短文案，例如：“正在解析堆栈中的命运分支...”

规则：

- 动画要服务于等待反馈，不应超过用户可接受的等待时长。
- 抽牌中不能重复点击抽牌按钮。

### 5.4 抽牌结果状态

触发条件：

- 系统完成抽牌和解读生成。

页面表现：

- 三张卡牌完成 3D 翻开。
- 解读区显示本次 Debug 占卜结果。
- 结果操作区显示重新抽牌、复制结果、生成截图。

用户可做：

- 查看三张牌。
- 阅读解读。
- 重新抽牌。
- 复制结果。
- 生成截图。

### 5.5 截图生成失败状态

触发条件：

- 浏览器截图能力失败。
- 目标结果区域渲染失败。
- 图片生成过程中发生异常。

页面表现：

- 显示短提示，例如：“仪式中断，请重新同步。”
- 保留当前牌面和解读结果。
- 用户可以重新尝试生成截图或复制结果。

规则：

- 失败不能清空当前抽牌结果。
- 失败提示不应遮挡三张牌和解读文本。

## 6. 截图生成范围

生成截图时应包含：

- 项目名 `Segfault Shrine`。
- 一句短标语或占卜标题。
- 用户输入报错的短摘录。
- 三张 Debug 塔罗牌。
- 三张牌名和关键词。
- Debug 占卜解读。

生成截图时不应包含：

- 输入框完整日志，避免截图过长。
- 操作按钮。
- 调试用信息。
- 未完成加载状态。

截图要求：

- 图片在社交聊天窗口中也能看清核心内容。
- 移动端截图区域需要避免过长，可以优先截取结果卡片和解读摘要。
- 若完整解读过长，截图中展示摘要，复制结果时保留完整文案。

## 7. 移动端适配

移动端目标：

- 用户可以用手机完成输入、抽牌、查看结果、复制和生成截图。

布局变化：

- 页面从桌面多列布局改为单列纵向布局。
- 标题区更紧凑，避免占满首屏。
- 输入框高度适当降低，给虚拟键盘留出空间。
- 三张卡牌从横向并列变为纵向堆叠、横向滑动或轮播。
- 解读区放在卡牌区之后。
- 操作按钮使用全宽或底部易触达布局。

交互变化：

- 不依赖 hover。
- 卡牌翻转、倾斜或查看重点信息使用 tap。
- 生成截图后提示用户保存或分享。

移动端注意：

- 底部操作区不能遮挡解读文本。
- 卡牌不能因为屏幕太窄导致文字溢出。
- 按钮文字必须完整显示。
- 截图生成范围需要适配窄屏。

## 8. MVP 之外的交互

以下内容不作为第一版必须实现：

- 卡牌大图详情弹窗。
- 卡牌图鉴页。
- 多种牌阵选择。
- 多种截图模板。
- 抽牌音效。
- 社交平台一键分享。
- 云端历史记录。
- 用户自定义牌库。

如果视觉工具生成了这些内容，可以作为设计灵感保留，但开发时不纳入 MVP。

## 9. 阶段验收标准

用户流程和页面结构确认后，应满足：

- 核心路径完整：输入报错、抽牌、看结果、复制或截图、重新抽牌。
- 页面只有一个核心应用页，没有超出 MVP 的页面。
- 每个主要模块都有明确目标和交互。
- 初始、空输入、抽牌中、结果、截图失败状态都有定义。
- 移动端核心流程可完成。
- 截图生成范围清楚。
- 没有引入登录、数据库、AI API 或真实代码分析能力。
