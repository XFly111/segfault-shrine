# Segfault Shrine MVP Roadmap

## MVP Must

第一版只保留以下功能。没有这些，项目的核心体验就不成立。

- 单页 Web 应用
  - 原因：MVP 的目标是快速完成一次 Debug 占卜，不需要多页面结构。

- 报错信息输入框
  - 原因：用户需要输入自己的错误信息，才能产生“这是为我占卜”的代入感。

- 抽牌按钮
  - 原因：这是核心操作入口，也是仪式感的触发点。

- 36 张本地 Debug 塔罗牌牌库
  - 原因：足够支撑随机性和重复游玩，同时仍然适合第一版维护。

- 每次抽出 3 张不同的牌
  - 原因：三张牌形成组合解读，比单张牌更有戏剧性，也符合 PRD 设定。

- 报错关键词影响抽牌
  - 原因：输入内容如果能影响结果，用户会觉得体验更“灵”。没有关键词命中时随机抽取。

- 三张牌代表三个不同 Bug 线索
  - 原因：比传统过去、现在、未来更贴近 Debug 场景。

- 3D 翻牌效果
  - 原因：这是项目的核心视觉记忆点，也是“抽牌”体验成立的关键。

- 本地模板生成神棍 Debug 解读
  - 原因：不接 API 也能完成完整体验，避免密钥、费用和网络依赖。

- 重新抽牌
  - 原因：用户会自然想多试几次，这是低成本提升可玩性的功能。

- 复制结果文案
  - 原因：方便用户把占卜结果发给朋友或群聊。

- 生成结果截图
  - 原因：这是第一版传播感的关键能力，符合用户想分享有趣结果的场景。

- 移动端基础适配
  - 原因：分享链接后很多用户会从手机打开，核心流程必须可用。

- 精致塔罗牌视觉基调
  - 原因：这是项目区别于普通随机文本生成器的主要气质。

## Should

这些功能重要，但可以在 MVP 可用后作为第二版增强。

- 更复杂的视觉特效组合
  - 原因：例如代码雨、魔法阵、粒子、扫描线都很适合项目，但第一版先保证卡牌和核心流程精致。

- 更丰富的关键词匹配规则
  - 原因：MVP 可以先覆盖常见报错类型，后续再扩充语言、框架和运行环境相关关键词。

- 多种解读语气
  - 原因：例如严肃神谕、毒舌前辈、赛博祭司。很有趣，但不是第一版核心闭环必需。

- 结果截图美化模板
  - 原因：MVP 只需要能生成可分享截图，后续再做不同海报样式。

- 抽牌音效和细节动效
  - 原因：能增强仪式感，但第一版可先用视觉完成主要体验。

- 本地历史记录
  - 原因：用户可能想回看之前抽到的结果，但不影响首次体验。

## Could

这些功能有价值，但暂时不影响主流程。

- 自定义牌阵名称
  - 原因：例如“生产环境三连”、“周五部署阵”。趣味强，但不是必要功能。

- 自定义用户名或项目名
  - 原因：可以让解读更个性化，但会增加表单复杂度。

- 一键复制分享链接
  - 原因：如果不接后端，持久化分享链接需要设计 URL 状态或本地编码，可以后续再做。

- 多语言支持
  - 原因：面向更广用户有价值，但第一版先聚焦中文程序员语境。

- 卡牌收藏或图鉴页
  - 原因：牌库内容值得展示，但会引入额外页面和状态。

- 自动保存截图到本地
  - 原因：浏览器兼容和权限体验需要额外处理，MVP 先提供生成截图能力。

## Later

这些功能留到项目验证后再考虑。

- 接入 AI API 生成解读
  - 原因：会引入密钥、费用、网络失败和安全问题，不适合第一版。

- 用户登录和账号系统
  - 原因：当前项目是轻量娱乐工具，不需要身份系统。

- 数据库和云端历史记录
  - 原因：MVP 没有必须保存到服务端的数据。

- 社区、点赞、评论、排行榜
  - 原因：会把项目从单页工具变成社区产品，范围过大。

- 后台管理牌库
  - 原因：第一版牌库可以直接在代码中维护。

- 支付和会员能力
  - 原因：与当前娱乐 MVP 无关。

- 真实代码分析和真实修复建议
  - 原因：这会改变项目定位，也会显著增加技术复杂度。

- 多人协作 Debug 房间
  - 原因：需要实时通信和房间状态，不适合 MVP。

## 第一版明确不做

- 不接 OpenAI 或其他 AI API。
- 不做登录、注册、用户系统。
- 不接数据库。
- 不做支付。
- 不做后台管理。
- 不做真实代码分析。
- 不做社区互动。
- 不做复杂历史记录。
- 不做多页面应用。
- 不做完整高保真设计系统后再开发。

## MVP 验收标准

- 页面打开后能看到项目名、输入区和抽牌入口。
- 用户可以输入任意报错或日志片段。
- 用户点击抽牌后，系统会抽出三张不同的牌。
- 输入命中预设关键词时，相关牌出现概率更高。
- 输入没有命中关键词时，系统仍可随机抽出三张牌。
- 三张牌有明确的 3D 翻开效果。
- 每张牌能展示牌名、关键词和简短说明。
- 系统会根据输入内容和三张牌生成一段 Debug 占卜解读。
- 用户可以重新抽牌。
- 用户可以复制结果文案。
- 用户可以生成结果截图。
- 桌面端布局完整，不遮挡主要内容。
- 移动端可以完成输入、抽牌、查看结果、复制和生成截图。
- 不需要真实 API key 或任何密钥即可运行。
- 项目可以本地运行并最终部署到公开访问地址。
